

理想之城（Ideal City）

愿景 → 工程语言转换文档（冻结版）

状态：冻结（Freeze）
本文档用于将已确定的世界愿景，翻译为工程可理解、可审计、不可越界的语义规则。
禁止在本文档基础上进行愿景优化、玩法设计或实现补全。

⸻

0. 角色与边界（必须接受）
	•	世界主权：在后端（世界裁决层），不在插件、不在 mod、不在算法平台。
	•	Copilot 职责：理解语义、识别差异、标注迁移路径；不设计玩法、不实现功能。
	•	技术约束：理想之城必须在 无任何科技 mod 的情况下完整运行；mod 仅是可插拔的表现/执行层。

⸻

1. 世界线定位（工程语义）
	•	世界线名称：理想之城（Ideal City）
	•	工程状态：Drift System 的 独立 fork（非并行扩展）
	•	兼容性要求：
	•	昆明湖 / 心悦宇宙 不要求保留
	•	关卡、剧情、叙事数据 允许整体替换
	•	不可破坏项（Drift 核心范式）：
	•	世界状态在后端裁决
	•	表现层仅执行补丁
	•	长期记忆/阶段推进不可被表现层绕过

⸻

2. 核心行为模型（工程化表述）

2.1 玩家行为（输入侧）
	•	玩家通过自然语言提出：
	•	面向社会/民生问题的创造方案
	•	描述逻辑关系、约束、目标
	•	输入不是命令、不是代码、不是直接建造指令

2.2 世界理解（抽象侧）
	•	玩家输入被抽象为一个逻辑对象：
	•	DeviceSpec（装置逻辑规格）
	•	DeviceSpec 是领域一等对象，与任务（Task）并行存在
	•	DeviceSpec 描述：
	•	问题意图
	•	世界约束
	•	逻辑步骤/关系
	•	成功/失败判据（语义级）

⚠️ 禁止：将自然语言直接转译为世界补丁、命令或实体建造

⸻

3. 世界裁决模型（不可替代）

3.1 裁决定义（工程语义）
	•	世界裁决 = 后端对 DeviceSpec 的审议结论
	•	裁决结果 只能是：
	•	接受（Accept）
	•	部分接受（Partial）
	•	拒绝（Reject）

3.2 裁决维度（非算法判分）

裁决依据 不是 OJ / 测试点 / AC/WA，而是：
	1.	逻辑成立性：方案是否闭合、有边界意识
	2.	世界约束一致性：是否违背社会/环境/阶段约束
	3.	长期影响：是否引入新的系统性风险

⚠️ 算法题 仅作为问题结构来源，不得参与裁决机制

3.3 裁决输出（与表现层解耦）
	•	裁决结果必须生成：
	•	结论（Accept / Partial / Reject）
	•	裁决解释（Reasoning / Evidence）
	•	裁决解释是可持久化数据，不是即时 UI 文案

⸻

4. 表现 / 执行层（可插拔）

4.1 表现层职责（允许）
	•	将裁决结果转化为：
	•	剧情节点
	•	世界补丁
	•	NPC 反馈
	•	场景变化

4.2 科技 mod 的合法位置
	•	mod 只能存在于：
	•	现象呈现层（题面/现象可视化）
	•	执行层（将已被接受的 DeviceSpec 实体化）
	•	mod 不得：
	•	参与裁决
	•	影响裁决结果
	•	在无裁决情况下执行逻辑

⚠️ 若无 mod，系统必须仍可运行（以叙事/状态变化形式完成反馈）

⸻

5. 数据与记忆（工程语义）
	•	DeviceSpec 与 裁决结果 必须：
	•	可被记录
	•	可被回溯
	•	可影响后续阶段/推荐/情绪
	•	现有：
	•	记忆系统
	•	情绪系统
	•	阶段推进
可被语义提升为：
	•	“世界裁决历史”的承载者

⸻

6. 禁止项（硬性红线）

以下实现一旦出现，即视为偏离愿景：
	1.	自然语言 → 直接生成机器/结构
	2.	使用 OJ / 算法判分作为裁决
	3.	mod 作为裁决主体或先于裁决执行
	4.	表现层（插件）承担逻辑判定
	5.	将 DeviceSpec 降级为一次性任务或剧情脚本

⸻

7. 迁移理解提示（给 Copilot）
	•	这是一次概念补齐，不是系统替换
	•	重点关注：
	•	领域抽象是否存在
	•	裁决语义是否独立
	•	表现层是否越界
	•	不需要：
	•	设计新玩法
	•	给出实现方案
	•	优化系统性能

⸻

8. 结束语（工程锚点）

理想之城不是“让玩家造机器”，
而是让创造成为一种可被世界裁决、记忆与承担后果的行为。

工程的责任，是守住这个裁决位置。
